## 三、第二阶段：超维度仓储中枢（规划中）

### 3.1 设计概念

**超维度仓储中枢**是 AE2Enhanced 模组的第二个大型多方块结构，定位是解决 AE2 网络中后期存储容量瓶颈问题。

- **核心卖点**：单结构提供 **BigInteger 级别**的存储容量，理论上无上限
- **兼容性目标**：物品、流体、气体（Mekanism）、源质（Thaumcraft）、魔力（Botania）等绝大多数 Mod 存储类型
- **性能策略**：数据**不写入 NBT/WorldSavedData**，而是持久化到**外部文件**，避免大容量存储导致的存档膨胀和加载卡顿
- **视觉体验**：以**高级感渲染**为核心亮点，结构本身即是一件视觉装置

### 3.3 核心架构

#### 3.3.1 无限存储系统

```
┌─────────────────────────────────────────────────────────────┐
│                    超维度仓储中枢 控制器                       │
│                     (TileEntity)                            │
├─────────────────────────────────────────────────────────────┤
│  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐  │
│  │  IStorage    │◄──►│  类型适配器   │◄──►│  外部文件层   │  │
│  │  Channel接口  │    │  (Adapter)   │    │  (File I/O)  │  │
│  └──────────────┘    └──────────────┘    └──────────────┘  │
│         ▲                                                   │
│         │ AE2 网络 (via ME接口)                             │
│         ▼                                                   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │              AE2-UEL Storage Grid                    │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

**关键设计点**：

1. **BigInteger 容量**
   - 所有容量计数器使用 `java.math.BigInteger`，彻底绕过 `int`/`long` 上限
   - AE2 内部的 `IAEStack` 数量字段为 `long`，需要自定义 `IMEInventoryHandler` 实现：对外查询返回 `BigInteger`，向 AE2 提交时按 `long` 分片或截断为 `Long.MAX_VALUE`
   - 实际实现策略：`getAvailableItems()` 返回的 `IAEItemStack` 数量上限为 `Long.MAX_VALUE`，超出部分在内部以 `BigInteger` 维护，取用时循环分片交付

2. **外部文件持久化（性能核心）**
   - **文件位置**：`<world>/ae2enhanced/storage/<nexus_uuid>.dat`
   - **文件格式**：压缩 NBT（`CompressedStreamTools`），版本化头部
     ```
     [4 bytes: 版本号 int]
     [NBT: 元数据（创建时间、模版本号、结构坐标）]
     [NBT: 索引表 Map<String, BigInteger>]
     [预留扩展区]
     ```
   - **版本化与向前兼容**：
     - 文件头部写入格式版本号（当前 v1）
     - 加载时若版本号低于当前，执行自动迁移逻辑
     - 预留未知 NBT 键，不认识的字段原样保留回写
     - **更新 mod 不会丢失玩家存储物品**
   - **索引结构**：内存中维护 `Map<StorageKey, BigInteger>` 热缓存，启动时全量加载索引；实际物品数据不需要额外索引，因为单个文件直接存储全部键值对
   - **写入策略**：**异步定时批量写入**
     - 内存中维护一个 `dirty` 标志集合
     - 每 5 秒（100 ticks）或结构卸载时，后台线程将 dirty 数据序列化写入文件
     - 写入前打快照，避免序列化过程中被修改
     - 写入失败时保留旧文件，写临时文件后原子重命名（`tmp → dat`）
   - **并发安全**：
     - 主线程：`insert`/`extract` 操作只修改内存中的 `ConcurrentHashMap`，标记 dirty
     - 后台线程：定时读取 dirty 快照，序列化写入磁盘
     - 无读写锁竞争，主线程永不等待 I/O

3. **类型兼容体系（适配器模式）**
   - 基类 `StorageAdapter<T>` 定义统一接口：`insert`、`extract`、`getAvailable`、`getCapacity`
   - 每种存储类型实现独立 Adapter：
     - `ItemStorageAdapter` — 物品（AE2 原生 `IItemStorageChannel`）
     - `FluidStorageAdapter` — 流体（AE2 流体存储频道或对接 `IFluidHandler`）
     - `GasStorageAdapter` — Mekanism 气体（对接 `IGasHandler`）
     - `AspectStorageAdapter` — 神秘时代源质（对接 `IEssentiaTransport` 或 `AspectList`）
     - `ManaStorageAdapter` — 植物魔法魔力（对接 `IManaReceiver` 或自定义协议）
   - 未识别的存储类型可通过配置文件或 API 由其他 Mod 扩展

#### 3.3.2 AE2 网络集成

- **接入方式**：**已确认 — ME 接口作为唯一网络接入点**
  - `TileHyperdimensionalMeInterface`：实现 `IGridHost` / `IActionHost`，持有独立 `AENetworkProxy`
  - 接口本身**不持有存储逻辑**，仅作为网络与控制器之间的桥梁
  - 控制器实现 `IStorageMonitorableAccessor` 或 `IMEInventoryHandler`，通过接口向 AE2 网络暴露存储能力
  - 结构形成后，接口的 `getGridNode()` 返回有效 node；结构解散后返回 `null`
- **单接口策略**：由于只有一个 ME 接口，不需要第一阶段的多接口竞争逻辑，逻辑大幅简化

#### 3.3.3 渲染系统（视觉核心）

> **用户确认**：21 格平面结构，**上方空气区域**提供视觉效果；控制器 GUI 为信息面板。

**核心视觉：超立方体（Tesseract）**
- **位置**：结构中心正上方（约 Y+1 ~ Y+3 区域）
- **主体**：一个**不断变化的超立方体**（四维立方体在三维空间的投影），或更复杂的几何图形（如超球体、克莱因瓶投影、多胞体旋转等）
- **动态表现**：
  - 持续慢速自转（双轴旋转模拟四维投影变化）
  - 边线/面片使用半透明发光纹理，带颜色呼吸效果
  - 内部可加入嵌套几何体或数据流粒子，暗示"无限维度存储"概念
  - 可考虑线框+实体混合渲染：外层线框脉冲，内层核心稳定发光
- **技术实现**：
  - TESR（`TileEntitySpecialRenderer`）绑定在中心控制器 Tile 上
  - 跨方块渲染范围：以控制器为中心，半径约 2~3 格的立方体空间
  - 使用 `GL11` 手动构建顶点（1.12.2 无现代 OpenGL），注意禁用深度写入或调整混合模式避免透明排序问题
  - 几何数据预计算：旋转矩阵和顶点坐标在 `onLoad` 时缓存，每帧只更新旋转角度，降低 CPU 开销

**外围特效（待细化）**

  - 轨道环（几何体公转效果）
  - 数据流粒子（从结构表面向超立方体汇聚，或从超立方体向外辐射）
  - 底部光环（结构平面边缘的地面投影/辉光）
- 具体设计留待后续迭代确定

**结构外壳反馈**
- CTM 或方块状态驱动的外壳纹理变化
- 根据存储负载（条目数、总物品量、网络活跃度）调整发光强度或色调

- 显示内容：在线状态、存储类型统计、总条目数、各类型占用比例、最近活动日志、渲染调试信息（可选）等

### 3.4 性能优化策略（核心原则：服务端 TPS 优先）

> **用户明确优先级：服务端 TPS 为最高优先级，客户端渲染可适当放宽。**

所有设计决策以满足服务端零负担为首要目标，客户端特效以视觉表现为先，基础优化兜底即可。

---

#### 服务端优化（最高优先级）

**内存优化**
- **热缓存限制**：内存中只保留索引表（`Map<StorageKey, BigInteger>`），不缓存实际物品栈对象
- **LRU 快照缓存**：`getAvailableItems()` 返回的 `IAEItemStack` 列表使用 LRU 缓存，5s 过期，避免 AE2 网络每 tick 全量查询导致大量对象创建
- **懒加载**：结构首次形成时才初始化文件句柄和缓存；未形成时控制器 Tile 为纯壳，不占用额外内存
- **引用复用**：同类型 `IAEItemStack` 查询结果复用同一对象实例（AE2 的 `AEItemStack` 支持 `copy()` 按需创建，优先返回只读视图）

**I/O 优化（零主线程阻塞）**
- **独立后台线程**：所有文件写入在独立 `ScheduledExecutorService` 中执行，主线程永不等待 I/O
- **批量合并**：5s 窗口期内多次修改合并为一次写入；单次写入数据量上限设为 1MB，超出则拆分多帧
- **原子写入**：先写 `.tmp` 文件，成功后重命名为 `.dat`，崩溃/强制关服不会损坏旧数据
- **写入队列背压**：若后台线程连续两次写入耗时 > 500ms，自动降频为 10s 间隔，避免磁盘瓶颈反向影响服务端
- **压缩**：使用 `CompressedStreamTools` 对 NBT 进行 GZip 压缩，减少磁盘 I/O 量

**AE2 网络优化（TPS 关键）**
- **零 Tick 扫描**：控制器 Tile 的 `update()` 中不做任何网络遍历或存储盘点，仅维护在线状态标志
- **被动响应模型**：`getAvailableItems()` 只在 AE2 请求时返回数据，不主动推送；内部使用预构建的不可变快照
- **差异通知**：使用 AE2 的 `IMEMonitorHandler` 机制，只在实际发生 insert/extract 时通知网络更新，避免无意义刷新
- **节流缓存**：每 tick 最多处理一次 AE2 全量查询，重复查询在 1 tick 内直接返回同一缓存快照（引用级复用）
- **分片交付**：向 AE2 提交超过 `Long.MAX_VALUE` 的物品时，按 `Long.MAX_VALUE` 分片返回，避免单次构造超大列表
- **无 Mixin 网络层**：除非绝对必要，不通过 Mixin 修改 AE2 的网络包处理逻辑，降低兼容性风险

**并发与线程安全**
- **主线程唯一写者**：只有服务端主线程修改内存中的存储数据；后台线程只读快照，序列化后写入磁盘
- **快照隔离**：定时写入前，主线程在 tick 末尾打一份 `Map` 浅拷贝快照传给后台线程，主线程继续处理下一 tick，两者无锁并发
- **无锁读路径**：AE2 网络的 `extract`/`insert` 查询走 `ConcurrentHashMap`，读操作无阻塞

---

#### 客户端优化（基础兜底，视觉优先）

- **视锥剔除**：TESR 中检查 `Frustum.isBoundingBoxInFrustum()`，不在视野内直接跳过
- **距离 LOD**（可放宽）：
  - `< 32 格`：完整特效（全息投影 + 粒子 + 动态光）
  - `32~64 格`：简化特效（降低粒子密度）
  - `> 64 格`：仅保留动态纹理/发光，关闭 TESR 和粒子
- **粒子上限**：单结构最多 128 个粒子（适度放宽），超出后停止生成
- **状态复用**：OpenGL 状态（纹理绑定、混合模式）在渲染帧内复用
- **禁用不必要的更新**：结构未形成时，控制器 Tile 不发送渲染更新包

> **FPS 约束放宽**：同屏多个结构的渲染负载可接受中等程度 FPS 下降（< 15%），不追求极致零开销。
